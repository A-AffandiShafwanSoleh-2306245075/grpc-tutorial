# Reflection

**1. What are the key differences between unary, server streaming, and bi-directional streaming RPC methods, and in what scenarios would each be most suitable?**

Unary RPC adalah pola komunikasi paling sederhana dimana client mengirim satu request dan menerima satu response, cocok digunakan untuk operasi sederhana seperti autentikasi user, mengambil satu data dari database, atau memproses pembayaran. Server Streaming RPC memungkinkan server mengirim banyak response untuk satu request dari client, cocok digunakan ketika server perlu mengirim data dalam jumlah besar seperti riwayat transaksi, live feed berita, atau update harga saham secara real-time. Bi-directional Streaming RPC memungkinkan client dan server saling mengirim banyak pesan secara bersamaan, paling cocok untuk aplikasi yang membutuhkan komunikasi dua arah secara real-time seperti aplikasi chat, game online, atau kolaborasi dokumen secara langsung.

**2. What are the potential security considerations involved in implementing a gRPC service in Rust, particularly regarding authentication, authorization, and data encryption?**

Dalam implementasi gRPC di Rust, terdapat beberapa pertimbangan keamanan penting. Pertama, untuk enkripsi data perlu menggunakan TLS/SSL agar data yang ditransmisikan tidak dapat disadap. Kedua, untuk autentikasi perlu mengimplementasikan mekanisme seperti JWT token atau mutual TLS agar hanya client yang terotorisasi yang dapat mengakses service. Ketiga, untuk otorisasi perlu memastikan setiap request divalidasi apakah user memiliki hak akses ke resource yang diminta. Keempat, perlu melindungi service dari serangan seperti DDoS dengan membatasi rate request. Dalam tutorial ini, aspek keamanan belum diimplementasikan sehingga service berjalan tanpa autentikasi apapun, yang tidak aman untuk lingkungan produksi.

**3. What are the potential challenges or issues that may arise when handling bidirectional streaming in Rust gRPC, especially in scenarios like chat applications?**

Beberapa tantangan dalam menangani bi-directional streaming di Rust gRPC antara lain: pertama, manajemen koneksi yang kompleks karena harus menangani banyak stream secara bersamaan yang dapat menyebabkan memory leak jika tidak dikelola dengan baik. Kedua, penanganan error yang rumit karena error dapat terjadi di sisi client maupun server kapan saja selama stream berlangsung. Ketiga, masalah backpressure dimana jika salah satu pihak mengirim pesan terlalu cepat dapat menyebabkan buffer penuh dan pesan hilang. Keempat, sinkronisasi state antara client dan server yang sulit terutama saat koneksi terputus tiba-tiba dan perlu melakukan reconnect.

**4. What are the advantages and disadvantages of using the tokio_stream::wrappers::ReceiverStream for streaming responses in Rust gRPC services?**

Keuntungan menggunakan ReceiverStream adalah integrasi yang mudah dengan ekosistem Tokio sehingga tidak perlu menulis boilerplate code yang banyak, mendukung async/await sehingga operasi non-blocking dapat dilakukan dengan efisien, dan memiliki mekanisme backpressure bawaan melalui bounded channel. Kerugiannya adalah ReceiverStream hanya mendukung pola single-producer yang membatasi fleksibilitas dalam skenario tertentu, ketergantungan pada ekosistem Tokio yang berarti sulit untuk diganti dengan runtime async lain, dan bounded channel dapat menyebabkan deadlock jika tidak dikonfigurasi dengan benar.

**5. In what ways could the Rust gRPC code be structured to facilitate code reuse and modularity, promoting maintainability and extensibility over time?**

Kode gRPC di Rust dapat distruktur dengan lebih modular dengan beberapa cara. Pertama, memisahkan setiap service ke dalam file atau module tersendiri misalnya payment_service.rs, transaction_service.rs, dan chat_service.rs agar lebih mudah dimaintain. Kedua, membuat trait atau interface yang dapat digunakan ulang untuk operasi umum seperti logging, error handling, dan validasi. Ketiga, menggunakan dependency injection agar setiap service tidak bergantung langsung pada implementasi konkret sehingga mudah diganti saat testing. Keempat, membuat layer middleware untuk menangani cross-cutting concerns seperti autentikasi dan rate limiting secara terpusat.

**6. In the MyPaymentService implementation, what additional steps might be necessary to handle more complex payment processing logic?**

Untuk menangani logika pemrosesan pembayaran yang lebih kompleks, beberapa langkah tambahan yang diperlukan antara lain: pertama, validasi input seperti memastikan amount lebih dari nol dan user_id valid sebelum memproses pembayaran. Kedua, integrasi dengan payment gateway eksternal seperti Midtrans atau Stripe untuk memproses pembayaran secara nyata. Ketiga, implementasi mekanisme idempotency agar request pembayaran yang sama tidak diproses dua kali. Keempat, menyimpan riwayat transaksi ke database untuk keperluan audit dan rekonsiliasi. Kelima, implementasi mekanisme rollback jika terjadi kegagalan di tengah proses pembayaran untuk menjaga konsistensi data.

**7. What impact does the adoption of gRPC as a communication protocol have on the overall architecture and design of distributed systems, particularly in terms of interoperability with other technologies and platforms?**

Adopsi gRPC memberikan dampak signifikan pada arsitektur sistem terdistribusi. Di sisi positif, gRPC mendorong desain berbasis kontrak melalui Protocol Buffers yang membuat antarmuka antar service lebih jelas dan terdefinisi dengan baik, serta mendukung code generation untuk berbagai bahasa pemrograman sehingga memudahkan interoperabilitas antar service yang ditulis dalam bahasa berbeda. Namun di sisi negatif, gRPC menggunakan HTTP/2 yang tidak didukung secara native oleh semua platform terutama browser web sehingga memerlukan proxy seperti gRPC-Web untuk komunikasi dengan frontend, dan kurva pembelajaran yang lebih tinggi dibandingkan REST API karena memerlukan pemahaman tentang Protocol Buffers dan konsep streaming.

**8. What are the advantages and disadvantages of using HTTP/2, the underlying protocol for gRPC, compared to HTTP/1.1 or HTTP/1.1 with WebSocket for REST APIs?**

HTTP/2 memiliki beberapa keunggulan dibanding HTTP/1.1 yaitu multiplexing yang memungkinkan banyak request dikirim dalam satu koneksi TCP secara bersamaan tanpa head-of-line blocking, header compression yang mengurangi overhead ukuran data yang dikirim, dan server push yang memungkinkan server mengirim data sebelum client memintanya. Namun HTTP/2 juga memiliki kekurangan yaitu kompleksitas implementasi yang lebih tinggi, tidak semua infrastruktur lama mendukung HTTP/2, dan debugging yang lebih sulit karena data dikirim dalam format binary bukan teks. Dibanding WebSocket, HTTP/2 lebih efisien untuk komunikasi request-response namun WebSocket lebih sederhana untuk implementasi komunikasi dua arah yang terus-menerus.

**9. How does the request-response model of REST APIs contrast with the bidirectional streaming capabilities of gRPC in terms of real-time communication and responsiveness?**

Model request-response pada REST API bersifat satu arah dimana client harus selalu menginisiasi komunikasi dan menunggu response dari server, sehingga untuk mendapatkan update real-time client harus melakukan polling secara berkala yang tidak efisien dan menambah beban server. Sebaliknya, bi-directional streaming pada gRPC memungkinkan client dan server saling mengirim pesan kapan saja tanpa harus menunggu pihak lain selesai, sehingga lebih responsif dan efisien untuk skenario real-time seperti aplikasi chat atau live dashboard. Namun REST API lebih mudah diimplementasikan, lebih mudah di-debug, dan didukung oleh hampir semua platform dan tool yang ada saat ini.

**10. What are the implications of the schema-based approach of gRPC, using Protocol Buffers, compared to the more flexible, schema-less nature of JSON in REST API payloads?**

Pendekatan berbasis schema pada gRPC menggunakan Protocol Buffers memberikan beberapa keuntungan seperti validasi data yang ketat sehingga mengurangi bug akibat format data yang tidak sesuai, ukuran data yang lebih kecil karena diserialisasi dalam format binary, dan performa serialisasi/deserialisasi yang lebih cepat dibanding JSON. Namun pendekatan ini juga memiliki kelemahan yaitu kurang fleksibel karena setiap perubahan struktur data memerlukan update pada file .proto dan regenerasi kode, serta lebih sulit dibaca manusia karena format binary tidak dapat dibaca langsung seperti JSON. Sebaliknya, JSON lebih fleksibel dan mudah dibaca manusia namun lebih lambat dan menghasilkan ukuran payload yang lebih besar dibanding Protocol Buffers.
